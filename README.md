# M365 Copilot Bridge

A two-part scaffold that lets you chat with **Microsoft 365 Copilot** from the
**VS Code Chat panel** by piggy-backing on your already-authenticated browser
session. No API key, no Azure OpenAI provisioning — the Chrome extension drives
the M365 Copilot web UI on your behalf and streams the response back into VS
Code over a localhost WebSocket.

> **Heads up — read this before you build on it.**
> This approach automates a licensed enterprise tool by scraping its UI. That is
> almost certainly against your employer's acceptable-use policy for that
> service. It is also inherently fragile: any UI change Microsoft ships to
> `copilot.cloud.microsoft` can break the selectors in `content.js`. Treat this
> as a personal-productivity experiment, not a production integration. The
> "proper" alternatives (Azure OpenAI, the `vscode.lm` Language Model API, or a
> sanctioned Copilot API) are noted at the bottom of this README.

## Architecture

```
   ┌──────────────────────────────────┐
   │            VS Code               │
   │  ┌────────────────────────────┐  │
   │  │  Chat panel  → @m365 ...   │  │
   │  └──────────────┬─────────────┘  │
   │                 │ ChatRequest    │
   │  ┌──────────────▼─────────────┐  │
   │  │ Extension: chatParticipant │  │
   │  │   ↕ pending-request map    │  │
   │  │ Extension: WS server       │  │
   │  │   ws://127.0.0.1:39847     │  │
   │  └──────────────┬─────────────┘  │
   └─────────────────┼────────────────┘
                     │ JSON frames
   ┌─────────────────▼────────────────┐
   │   Chrome (MV3 service worker)    │
   │     background.js — WS client    │
   │            ↕ chrome.tabs         │
   │     content.js (injected into    │
   │     copilot.cloud.microsoft)     │
   │     • types into the composer    │
   │     • clicks send                │
   │     • MutationObserver on the    │
   │       last assistant message     │
   │     • streams text deltas back   │
   └──────────────────────────────────┘
```

## Wire protocol

JSON, line-delimited, over a single WebSocket. `id` is a UUID generated by the
VS Code side; every response frame for a prompt carries the same `id`.

**VS Code → Chrome**

```jsonc
{ "type": "prompt", "id": "uuid", "text": "user prompt..." }
{ "type": "cancel", "id": "uuid" }                 // user pressed Stop
```

**Chrome → VS Code**

```jsonc
{ "type": "hello", "role": "browser" }             // sent on connect
{ "type": "response.delta", "id": "uuid", "text": "incremental text" }
{ "type": "response.done",  "id": "uuid", "text": "final full text" }
{ "type": "response.error", "id": "uuid", "message": "..." }
```

Deltas carry the **full text seen so far**, not a token-by-token diff — the VS
Code side computes the diff and appends only the new tail to the chat stream.
This keeps the scraper trivial: it just emits the current `innerText` of the
assistant bubble on every mutation.

## Setup

### Prerequisites
- VS Code ≥ 1.95 (Chat API GA)
- Node 18+ (for building the extension)
- Chrome / Edge (Chromium-based, MV3)
- A logged-in M365 Copilot session in that browser

### Install the VS Code extension (dev mode)
```bash
cd vscode-extension
npm install
npm run compile
# Then in VS Code: F5 to launch an Extension Development Host
```

### Install the Chrome extension (unpacked)
1. Open `chrome://extensions`
2. Toggle **Developer mode** on
3. Click **Load unpacked** → select the `chrome-extension/` folder
4. Pin the extension; its icon shows the WS connection state

### Use it
1. Open `https://copilot.cloud.microsoft` in the browser where you installed the
   extension, sign in if prompted, and leave the tab open.
2. In VS Code, open the Chat panel and type `@m365 hello, are you there?`
3. The prompt is sent over WS → Chrome forwards it → the content script types
   it into the composer and clicks send → the streaming response shows up in
   the VS Code chat panel.

## Configuration

| Setting | Default | Notes |
| --- | --- | --- |
| `m365Bridge.port` | `39847` | Localhost-only. Change if the port is taken. |
| `m365Bridge.requestTimeoutMs` | `120000` | Per-prompt ceiling. |
| `m365Bridge.newChatPerRequest` | `false` | If true, content script clicks "New chat" before each send. |

## When (not if) it breaks

The DOM selectors in `chrome-extension/content.js` are the soft underbelly.
When Microsoft re-skins the Copilot UI, the scraper will start silently doing
nothing, or grabbing the wrong message. To debug:

1. Open the M365 Copilot tab, then **DevTools → Console**. The content script
   logs every step prefixed with `[m365-bridge]`.
2. Inspect the composer textarea / contenteditable — update `SELECTORS.input`.
3. Inspect the send button — update `SELECTORS.sendButton`.
4. Inspect a finished assistant message — update `SELECTORS.assistantBubble`
   and `SELECTORS.doneIndicator` (the "Stop generating" → "Regenerate"
   transition is usually the clearest "done" signal).

Each selector in `content.js` is an **array of candidates** tried in order;
add new candidates rather than replacing — this lets the extension survive
A/B-tested UI variants.

## Security notes (do these before daily use)

The scaffold binds the WS server to `127.0.0.1` only, but any process on your
machine can connect to that port. Two hardening steps worth doing:

1. **Shared token.** Generate a random string, paste it into both the VS Code
   setting `m365Bridge.sharedToken` and the Chrome extension's options page.
   The WS handshake rejects connections that don't present it.
2. **One-tab lock.** The Chrome extension currently forwards to the first
   matching tab. Tighten it to a specific tab id chosen via the popup, so a
   stray Copilot tab can't get hijacked.

Both are flagged with `TODO(hardening)` in the code.

## Why this is the wrong long-term answer

Stated honestly, in priority order:

1. **`vscode.lm` Language Model API.** If you have any Copilot license on the
   same machine, VS Code already exposes those models programmatically. A 100-
   line extension can list them with `vscode.lm.selectChatModels()` and call
   them — no scraping, no Chrome, no policy risk.
2. **Azure OpenAI.** If your tenant has an Azure OpenAI deployment, ask your
   platform team for an endpoint + key. The same chat-participant code in this
   repo points at a real API instead of a WebSocket.
3. **GitHub Copilot Chat.** Already exists, your org may already own it.

This repo is the route to take when none of the above are available to you and
you've decided the policy trade-off is acceptable.
